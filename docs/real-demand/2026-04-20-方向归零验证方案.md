# MoneyPrinter 方向归零验证方案

日期：`2026-04-20`

## 为什么要补这份文档

当前主线方向虽然已经通过《真需求》的框架做过一轮判断，但它还有一个关键缺口：

- 方向并不是从一手用户访谈、问卷、连续样本里长出来的
- 更像是：已有站点内容 -> AI 总结 -> 二次推导出当前创业方向

这不代表方向错误。

但它意味着：

> 现在必须把“方向判断”升级为“方向归零验证”，用一套可以证伪的流程，判断这条线到底是独立可成立的业务，还是只能作为更大服务中的一个模块。

## 当前最需要防的 5 个误判

### 1. 内容合理，不等于需求强

- 文章、清单、SOP 看起来都成立
- 不代表目标用户会继续走、会留关系、会付费

### 2. 问题存在，不等于需要一个新产品

即使 deployment / hardening 的问题真实存在，用户也可能已经被下面这些替代方案充分满足：

- GitHub README / docs
- issue / discussion
- 博客 / YouTube 教程
- 朋友 / 群友 / 顾问
- 托管平台 / managed service
- 通用 AI 对话

### 3. 免费点击，不等于付费意图

- 文章点击
- checklist 领取

都只能说明兴趣。

它们不能自动说明：

- 会继续问
- 会发真实项目过来
- 会为更深判断付费

### 4. AI 能帮忙，不等于 AI 已经足够

通用 AI 是当前最强的替代方案之一。

真正需要验证的是：

- 用户在哪一步会求助 AI
- AI 帮到了哪里
- AI 又卡在了哪里
- 我们是否正好补那个缺口

### 5. 独立数字资产不成立，不等于能力没价值

这条能力线有三种可能：

1. 纯 checklist / SOP 就能独立成立
2. 更适合作为 `项目级判断服务` 成立
3. 只能作为更完整 deployment / launch guidance 服务里的一个模块成立

## 这轮验证要回答的 6 个问题

### 1. 这个问题是否高频且高痛

要验证的是：

- “值不值得部署”
- “本地能跑但服务器不行”
- “哪些风险必须先处理”

这些问题是否在真实场景中反复出现。

### 2. 现有替代方案是否明显不够

重点看：

- 太碎
- 太泛
- 没顺序
- 没 go / no-go 判断
- 没有项目级上下文

### 3. 用户是否愿意为“判断和顺序”继续前进

继续前进包括：

- 留邮箱
- 回复
- 发项目链接
- 约时间
- 付小钱

### 4. 用户是否信任我们来做这件事

deployment / hardening 的建议很吃信任。

必须验证：

- 用户是否把我们视为“可承担后果的判断层”
- 还是只当成“又一个教程作者”

### 5. 这是不是独立产品

必须分清：

- 用户想要的是一份 checklist / SOP
- 还是更想要一个项目级判断服务
- 或者一套更完整服务

### 6. 我们是否有相对 AI 和现有替代方案的明显优势

要验证的不是“我们也能回答”。

而是：

- 我们是不是更准
- 更有顺序
- 更可信
- 更可复用

## 替代方案地图

这轮研究不要只做“竞争对手名单”。

至少要按下面 6 类建立替代方案地图。

### 1. 免费静态替代

- README
- 官方 docs
- 博客
- YouTube 教程

### 2. 免费互动替代

- GitHub issue / discussion
- 论坛答主
- 朋友 / 群友

### 3. 通用 AI 替代

- ChatGPT
- Claude
- Gemini
- Perplexity
- AI + README / issue 的组合用法

### 4. 专业服务替代

- DevOps 顾问
- 安全顾问
- deployment audit
- managed service

### 5. 数字资产替代

- checklist
- SOP
- template
- 课程
- toolkit

### 6. 放弃型替代

- 先不部署
- 改用 SaaS
- 改用更熟悉的项目

## 当前最值得单独研究的替代方案：通用 AI

通用 AI 是最需要单独研究的一条线。

因为它既能解决问题，也会压缩任何“泛知识型产品”的空间。

### 必须重点验证的 AI 弱点

#### 1. 上下文不完整

- 是否看不到真实 issue 密度
- 是否忽略维护状态
- 是否忽略默认暴露面

#### 2. 建议过于泛化

- 是否只给通用安全建议
- 但没有结合具体项目部署路径

#### 3. 缺少明确判断

- 不敢说值不值得部署
- 不敢给 go / no-go 结论
- 不敢说哪 3 个风险必须先做

#### 4. 缺少顺序和优先级

- 会列项
- 但不一定告诉用户先做什么、后做什么、哪些可以先不做

#### 5. 信任不足

- 用户会参考
- 但关键上线动作未必敢完全照做

#### 6. 结果不可复用

- 一次对话有帮助
- 但很难直接沉淀为团队 SOP、模板或项目判断记录

## 证据收集的 4 条主线

### 主线 1：真实问题证据库

去下面这些场景收集一手样本：

- GitHub issue / discussion
- Reddit
- V2EX
- 知乎
- Hacker News
- Stack Overflow
- Discord / forum / 社群

每条证据至少记录：

- `sample_date`
- `channel`
- `source_url_or_person`
- `user_quote`
- `scenario`
- `current_blocker`
- `feared_outcome`
- `current_workaround`
- `current_ai_tools_used`
- `current_ai_limitations`
- `requested_next_step`
- `matched_asset`
- `notes`

### 主线 2：替代方案地图

每个替代方案至少记录：

- 名称
- 类型
- 服务对象
- 使用时刻
- 核心承诺
- 产品形式
- CTA
- 价格
- 证据类型
- 常见抱怨
- 我们可能的应对

### 主线 3：用户访谈

目标不是问态度。

目标是追问最近一次真实经历。

每次访谈优先问：

- 最近一次你差点部署或已经部署的是什么项目
- 你当时最不确定的是什么
- 你最怕什么后果
- 你去哪里找答案
- 你是否用过 AI
- AI 帮到了哪一步
- AI 不够的地方是什么
- 如果有人帮你做“值不值得部署 + 先做什么”的判断，你会不会继续走
- 你更想要 checklist、项目判断、还是更完整服务

### 主线 4：最小实验

这轮实验至少要跑下面 4 个。

#### 实验 A：免费资源页

验证：

- 是否有人愿意从文章进入 checklist
- 是否有人愿意提交邮箱

#### 实验 B：低价意向页

验证：

- 是否有人愿意为更深版本付费或预约

建议测试物：

- `开源 AI 项目上线与基础加固 SOP`
- 或 `项目部署前判断服务`

#### 实验 C：手工 concierge 判断

动作：

- 让用户发项目链接
- 手工给出：
  - go / no-go
  - top risks
  - first 5 steps
  - postpone list

验证：

- 用户是否觉得这比自己看 README、问 AI 更值

#### 实验 D：AI-vs-我们 对照实验

选 `10-20` 个真实项目，产出两版：

- A 版：通用 AI 输出
- B 版：我们的方法输出

让目标用户盲评：

- 哪个更适合真正上线前使用
- 哪个更值得收藏 / 转发给同事
- 哪个更像“可承担后果的判断”
- 哪个更值得付费

## 本轮必须写清楚的证伪标准

如果下面信号连续出现，就要暂停这条线的进一步复杂化：

- 看了大量真实问题后，重复痛点仍很弱
- 大多数用户认为 README / issue / AI 已经够用
- 免费资源有点击，但几乎没人提交邮箱
- 有邮箱，但几乎没人愿意发真实项目来求判断
- 低价测试无人下单，也无人预约

出现这种情况时，不要再用“多写几篇内容试试”自我安慰。

应该明确回到 3 个判断：

1. 需求弱
2. 信任不足
3. 这项能力必须嵌入更大服务里才成立

## 这轮验证后可能出现的 3 种方向

### 方向 A：独立数字资产成立

信号：

- checklist / SOP 本身有稳定领取和付费意向

下一步：

- 强化数字资产路径

### 方向 B：项目级判断服务更成立

信号：

- 用户更愿意发项目链接求判断
- 更愿意买“判断和顺序”

下一步：

- 把主产品形态收敛到 `项目级 deployment judgment`

### 方向 C：只能作为更完整服务的模块成立

信号：

- 用户觉得这项能力有价值
- 但只有在更系统的服务里才愿意买

下一步：

- 把它当成更大服务中的核心模块，而不是单独产品

## 当前推荐执行顺序

1. 先收集真实问题和替代方案样本
2. 再访谈目标用户，确认 AI 和现有方案是否足够
3. 再跑手工 concierge 与低价意向测试
4. 最后才决定：
- 继续做纯数字资产
- 转向项目级判断服务
- 还是把这条能力嵌进更系统的服务

## 最小结论

当前最需要验证的不是：

- “这个主题是不是看起来像真需求”

而是：

> 当用户已经拥有 README、issue、博客、朋友和通用 AI 时，他是否仍然需要一个更可靠、更有顺序、更可复用的 deployment judgment 层。

如果答案是“是”，这条线值得继续。

如果答案是“否”，就不要把更多执行预算压在这条独立资产线上。
