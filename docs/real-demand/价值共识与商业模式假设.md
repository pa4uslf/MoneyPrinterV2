# 价值共识与商业模式假设

日期：`2026-05-19`

这份文档补足 `MoneyPrinterV2` 当前真需求验证里的一个关键缺口：

> 找到需求以后，还必须解释为什么客户要选择我们、我们存在的价值在哪里、如何与客户形成共识，以及这件事最终更适合变成哪种商业 / 盈利模式。

它不是新战略，也不是销售话术库。

它的作用是给后续选题、分发、访谈、AI-vs-我们实验、SOP 和服务判断增加一个门槛：

> 不允许把手里已有的 checklist、SOP、文章或 AI workflow 强行推给客户，必须先证明它确实替代了客户当前低效、碎片化或高风险的解决方式。

## 1. 当前要防的误判

### 误判一：有需求，不等于客户必须选我们

即使用户真的有 deployment / self-hosting / launch-readiness 问题，他也可能已经被下面这些方案满足：

- README / docs
- GitHub issue / discussion
- 博客 / YouTube 教程
- 朋友 / 群友
- ChatGPT / Claude / Gemini / Perplexity
- 顾问 / DevOps 服务
- managed service
- 暂时不部署

所以后续不能只问：

- 这个问题是否存在？

还必须问：

- 用户现在靠什么解决？
- 这些方案为什么不够？
- 我们凭什么比这些方案更适合这个具体时刻？

### 误判二：我们能做，不等于客户要买

Frank 当前可能有这些能力：

- 读项目和文档比较快
- 能把混乱信息整理成 checklist / SOP
- 能识别基础部署、暴露面、鉴权、回滚、运维边界风险
- 能用 AI 加速资料提炼和输出整理

但这些是内部能力，不是客户价值。

客户价值必须改写成：

- 少浪费一次部署试错
- 少踩一次公网暴露或配置事故
- 少在 README、issue、AI 回答之间来回拼答案
- 更快知道先做什么、后做什么、什么可以先不做
- 能把判断结果复用成团队 SOP、上线检查或执行记录

### 误判三：共识没形成，不要用更强销售解决

如果客户不继续走 checklist、项目判断、SOP 或服务，不要默认是销售话术不够强。

更可能是共识还没形成：

- 他没有觉得你说的是他的具体场景
- 他没有觉得现有方案真的不够
- 他没有相信你能承担判断后果
- 他没有看出 checklist / SOP / 项目判断之间的差异
- 他不知道下一步动作成本有多低、风险有多小

此时应该补证据、补场景、补对照，而不是把 CTA 写得更硬。

## 2. 我们存在的候选价值

当前最值得验证的价值不是“部署教程”，而是：

> 在用户已经能自己折腾、也能问 AI / 看 README 的情况下，提供一个更有顺序、更有边界、更能承担上线前后果的项目判断层。

这个价值可以拆成四类。

### 1. 判断价值

回答：

- 这个项目值不值得部署？
- 当前条件下能不能推进到第一版可用？
- 哪些风险足以影响 go / no-go？

适合形态：

- 项目级判断报告
- 低价 deployment readiness review
- concierge 判断服务

### 2. 顺序价值

回答：

- 先做什么？
- 后做什么？
- 哪些可以等到第一版之后？

适合形态：

- SOP
- playbook
- first 5 steps
- checklist plus execution guide

### 3. 风险收敛价值

回答：

- 哪些公网暴露、鉴权、密钥、回滚、备份、监控问题最容易先出事？
- 哪些风险第一版必须收住？
- 哪些风险可以标记但暂缓？

适合形态：

- hardening checklist
- launch readiness audit
- risk-priority memo

### 4. 可复用价值

回答：

- 这次判断能不能沉淀为下一次部署复用的材料？
- 能不能给团队、未来的自己、或客户交代？

适合形态：

- 模板包
- SOP 包
- Obsidian / Notion / Markdown 资产
- 团队上线检查记录

## 3. 为什么客户可能选择我们

后续任何内容或 offer 都不能只写“我们帮你部署 / 我们给你清单”。

必须至少命中下面一个差异点，并尽量用真实证据验证。

### 差异点 A：比通用 AI 更有项目上下文

要验证：

- AI 是否忽略 issue 密度、维护状态、默认暴露面、部署路径差异？
- 我们是否能把 README、issue、配置、风险和上线顺序合在一起判断？

证据来源：

- `AI对照实验模板.md`
- 真实项目样本
- 盲评结果

### 差异点 B：比 README / 教程更有顺序

要验证：

- README 是否只讲怎么跑，不讲是否该上线？
- 教程是否只讲局部步骤，不讲 first 5 steps / what can wait？
- 我们是否能给出更清楚的执行顺序？

证据来源：

- 用户访谈
- 替代方案地图
- 真实问题样本

### 差异点 C：比社区问答更可复用

要验证：

- issue / discussion 是否能解决一次问题，但很难沉淀成团队可复用 SOP？
- 我们是否能把零散问答整理成 checklist、判断报告或执行记录？

证据来源：

- demand evidence sample
- 项目级判断输出
- 用户是否愿意保存、转发、发给同事

### 差异点 D：比顾问服务更低摩擦

要验证：

- 用户是否不想一上来约咨询或买完整服务？
- 低价 SOP、项目判断或模板是否足以让用户先继续一步？

证据来源：

- 访谈中的 preferred_format
- would_pay_for_deeper_help
- 小额付费实验

## 4. 如何与客户形成共识

共识不是一次性靠首页文案完成。

当前更合理的共识链路是：

1. **场景共识**
   - 让用户觉得“这是我正在经历的问题”
   - 例：本地能跑，但 Docker / VPS / 公网一上就不稳

2. **代价共识**
   - 让用户意识到继续乱试会带来时间、返工、暴露或维护成本

3. **替代方案共识**
   - 承认 README、issue、AI、朋友都能帮一部分
   - 但指出它们在哪个具体时刻不够

4. **判断层共识**
   - 让用户理解我们不是又一个教程作者
   - 我们提供的是 go / no-go、top risks、first steps、what can wait

5. **下一步共识**
   - 让用户知道下一步很轻：
     - 领 checklist
     - 发一个项目链接
     - 看一份样例判断
     - 加入 SOP waitlist
     - 买一个低价 SOP / 模板

后续任何文章、资源页、邮件或 offer，如果跳过前四层直接进入销售，都容易变成“拿着锤子找钉子”。

## 5. 商业 / 盈利模式分流

当前不要提前假设一定是数字产品。

真实需求验证后，至少可能分成三条路线。

### 路线 A：独立数字资产成立

成立信号：

- checklist 有稳定领取和确认
- 多个用户问更完整版本
- 用户愿意为 SOP / 模板付小钱
- 问题重复性高，不需要每次深度定制

可能产品：

- paid SOP
- template pack
- launch checklist bundle
- risk-priority playbook

主要风险：

- 如果用户真正要的是项目级判断，纯 SOP 会显得太泛。

### 路线 B：项目级判断服务更成立

成立信号：

- 用户更愿意发项目链接
- 用户问“这个项目我该不该部署”
- 用户想要 go / no-go、top risks、first 5 steps，而不是只要一份文档
- 用户愿意为一次判断或审查付费

可能产品：

- deployment readiness review
- open-source project launch judgment
- 低价项目判断报告
- checklist + human/AI-assisted review

主要风险：

- 容易滑回卖时间，需要限制范围、固定输入、固定输出和固定价格。

### 路线 C：只能作为更完整服务的模块成立

成立信号：

- 用户觉得判断有价值，但不愿单独买
- 用户真正要的是从部署、配置、加固到上线的一整段陪跑
- 项目差异太大，难以产品化成低价资产

可能产品：

- launch guidance package
- bootstrap implementation service
- deployment / hardening bundle

主要风险：

- 容易变成普通服务商，所以必须把交付过程继续沉淀成资产、模板和案例。

## 6. 后续记录时必须补的字段

无论写进 `demand-evidence-samples.csv`、访谈记录、替代方案地图，还是分发日志，后续都应尽量补上这些判断字段：

- `why_not_current_solution`
  - 当前替代方案为什么不够
- `why_us_now`
  - 为什么这个时刻可能需要我们
- `consensus_stage`
  - 当前共识停在哪一层：场景 / 代价 / 替代方案 / 判断层 / 下一步
- `preferred_business_model`
  - 更像数字资产、项目判断服务，还是完整服务模块
- `must_choose_us_reason`
  - 如果用户必须选择我们，最强理由是什么
- `evidence_strength`
  - weak / medium / strong
- `current_spend_or_labor_cost`
  - 用户已经为这个问题付出的金钱、时间、人力或外包成本
- `cost_of_inaction`
  - 不解决会损失什么，例如收入、工期、客户、尾款、业务连续性或安全风险
- `budget_source`
  - 这笔钱或时间可能来自哪个预算或资源池
- `trigger_event`
  - 为什么是现在必须解决
- `buyer_vs_user`
  - 使用者和付款者是否同一个人
- `replacement_target`
  - 实际替代的是人工、外包、顾问、SaaS、摸索时间，还是只满足好奇
- `economic_force`
  - curiosity / efficiency / revenue / risk / survival

这些字段不用一次性重构所有 CSV。

但从现在开始，新增样本和访谈至少要在 notes 里回答：

> 为什么不是 README、issue、AI、朋友或顾问就够了？这个问题有没有明确经济强制力？

## 7. 当前下一步

截至 `2026-05-19`，这个项目不应该把重点放在更强包装已有资产。

更应该做：

1. 补真实需求样本时，同时记录当前替代方案、`why_not_current_solution` 和经济强制力字段
2. 做 AI-vs-我们 对照实验，验证 `why_us_now`
3. 访谈时明确追问 preferred_format 和 preferred_business_model
4. 从样本里判断到底是：
   - checklist / SOP 独立成立
   - 项目级判断服务更成立
   - 只能作为更完整服务里的模块成立
5. 只有共识链路跑通后，再更新销售页、SOP、支付或服务报价

核心判断句：

> 不是先证明我们能卖什么，而是先证明客户在已有替代方案之外，仍然需要我们提供一个更可靠的判断层。
