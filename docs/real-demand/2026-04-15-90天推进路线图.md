# MoneyPrinter 90 天推进路线图

起始日期：`2026-04-15`  
结束日期：`2026-07-14`

## 直接结论

未来 90 天的核心目标不是“把系统做大”，而是：

> 用最小执行成本，验证 deployment 主题簇是否真的能持续产出点击、关系和小额付费。

这 90 天应该分三段推进：

1. `先补真需求样本`
2. `再补漏斗证据`
3. `最后才测付费`

## 阶段一：2026-04-15 到 2026-05-14

主题：

- `需求识别 + 漏斗跑通`

### 本阶段目标

- 不再只靠内部文档判断需求
- 给 deployment 主题簇补第一批真实样本
- 验证 checklist 承接是否真的成立
- 判断 README / issue / AI 等现有替代方案是否已经足够好

### 必做动作

#### 1. 补足真实需求样本

最少完成：

- `10` 个来自公开问题现场的真实需求样本
- `5` 个来自已有关系链或私下交流的真实需求样本
- `20` 个带有“当前替代方案”信息的样本，其中至少 `10` 个明确记录是否使用过 AI

每条样本至少记录：

- 用户原话
- 场景
- 最怕什么
- 当前怎么解决
- 是否使用 AI / 用了什么 AI
- AI 或现有替代方案哪里不够
- 有没有更深版本需求

建议记录位置：

- 先写进你自己的知识库
- 再把提炼后的结论回写到项目文档或选题判断里

#### 2. 执行真实分发动作

最少完成：

- `5` 个真实分发动作

优先顺序：

1. GitHub 高相关 thread
2. 关系链定向发送
3. 低频 email update
4. 1 条轻量短帖

记录位置：

- `docs/distribution-log.csv`

#### 3. 完成 Batch 1 的前两篇主文验证

必须优先推进：

- `部署一个 GitHub 项目前，最应该先确认的 10 件事`
- `为什么很多开源 AI 项目“本地能跑”，一上服务器就出问题？`

本阶段不急着大量扩写新题。

#### 4. 检查 checklist 漏斗是否真的在工作

必须确认：

- 文章到 checklist 的点击是否出现
- checklist 页面是否有领取 / unlock
- 是否有确认订阅
- 是否有人回复或追问

#### 5. 建立替代方案地图

最少完成：

- `15` 个替代方案条目

至少覆盖：

- README / docs
- GitHub issue / discussion
- 博客 / YouTube
- 朋友 / 群友 / 顾问
- managed service
- 通用 AI

目标不是列公司名，而是判断：

- 用户现在实际上靠什么解决
- 哪些替代方案最常被使用
- 哪些替代方案最容易让用户不满

记录位置：

- `docs/topic-ledger.csv`

### 本阶段完成标准

满足下面大部分，才算阶段一完成：

- `5` 个真实分发动作已执行
- `15` 个真实需求样本已记录
- 替代方案地图已完成第一版
- 两篇主文都进入真实外部场景
- checklist 有真实点击或领取
- 至少出现 `2` 条以上深度反馈或追问

如果这些没有发生，不要推进更多资产层复杂化。

## 阶段二：2026-05-15 到 2026-06-14

主题：

- `收敛定位 + 提高共识`

### 本阶段目标

- 从样本里找出重复出现的痛点表达
- 收窄 ICP
- 把文章、资源页、邮件文案对齐到同一种真实语言
- 判断我们相对 AI 和现有替代方案的真实差异点

### 必做动作

#### 1. 收敛到 1 个主 ICP + 1 个次 ICP

主 ICP 建议优先在下面两类里选一个：

- 会自己折腾但怕上线事故的独立开发者
- 小团队里的兼职技术负责人

如果两类都成立，也必须明确：

- 谁是主用户
- 谁是顺带用户

#### 2. 用真实原话改写文案

优先改写：

- 文章标题
- checklist 首屏
- 成功页文案
- welcome 邮件

改写原则：

- 贴近用户原话
- 强调顺序感、暴露面、返工成本、保障感
- 少讲“知识很多”，多讲“少踩大坑”

#### 3. 决定第三篇桥梁文是否进入主线

只有下面条件出现，才把第三篇桥梁文提到更高优先级：

- checklist 已开始带来点击或领取
- 用户开始问“哪些风险先处理”
- 免费清单已经无法完整回答重复问题

#### 4. 整理 SOP 候选章节

不是正式上线 SOP。

只做一件事：

- 把真实重复问题整理成 SOP 最小章节骨架

优先保留：

- 第一版上线顺序
- 必做 vs 可后补
- 公网暴露 / 鉴权 / 密钥 / 回滚
- 常见环境差异与稳定性问题

#### 5. 跑 AI-vs-我们 对照实验

最少完成：

- `10` 个真实项目的对照样本

每个样本至少产出两版：

- A 版：通用 AI 基于现有公开信息给出的 deployment / hardening 建议
- B 版：我们给出的 go / no-go、top risks、first 5 steps、postpone list

验证目标：

- 我们是否真的比 AI 更有顺序
- 是否更像“判断层”而不是“泛教程层”
- 是否值得继续往付费方向推进

### 本阶段完成标准

- 主 ICP 已收敛
- 至少 `3` 个重复出现的高频痛点被确认
- checklist 文案已按真实语言改过一轮
- 至少 `1` 个相对 AI 更强的差异点被确认，或者明确确认目前没有明显优势
- topic ledger 不再只是 seed 状态
- 已经能判断卡点更偏流量、承接还是共识

## 阶段三：2026-06-15 到 2026-07-14

主题：

- `小额付费验证`

### 本阶段目标

- 不求规模化营收
- 只求验证“有没有人愿意为更完整的执行顺序和风险收敛付钱”
- 同时验证这是不是独立数字资产，还是更适合作为项目级判断服务

### 推进前门槛

下面这些信号至少满足大部分，才允许推进 SOP 上线实验：

- 多篇内容带来了 checklist 点击
- checklist 有稳定 unlock / 领取
- 有确认订阅
- 有至少 `3` 次更深版本追问
- 有重复出现的问题集合，足够支撑 SOP 章节

### 必做动作

#### 1. 做一个最小 SOP 实验版

第一版只解决：

- 从“准备部署”到“第一版可用”的执行顺序
- 明显风险的优先收敛

不要试图一次做成大全。

#### 2. 做轻量价格实验

建议区间：

- `¥39 - ¥99`

目标不是利润最大化，而是验证：

- 用户是否愿意为更深执行资产付费
- 哪种承接路径更有效

补充要求：

- 至少同时测试两种形态：
  - 纯 SOP / 数字资产
  - 项目级 deployment judgment / 轻服务

#### 3. 销售路径只走轻路径

优先路径：

- 文章 -> checklist -> email / success page -> SOP

当前仍然不建议：

- 首页强卖
- 全站到处挂付费 CTA
- 提前把重心切到 `shop.fzhang.dev`

### 本阶段完成标准

以下任一出现，就说明这条线值得继续：

- 有真实 SOP 购买
- 有真实项目判断服务预约或愿意发项目链接
- 有明确付费意向回复
- 有用户愿意进入 waitlist / 预热页
- checklist 用户明显往更深资产走

如果没有出现，不要硬推。

要回头检查：

- 免费资产是否已经真的解决前置问题
- 用户是否真的卡在“更完整版本”
- 还是其实卡点仍然在流量或承接
- 还是其实用户更想买“判断服务”而不是“数字资产”

## 90 天内明确不要做的事

- 不要再开新的大主题簇
- 不要把主线切去泛 AI 内容
- 不要把服务销售重新提到主优先级
- 不要先搞复杂支付与后台
- 不要为了“系统完整”做过多自动化
- 不要再靠写更多战略文档替代真实验证

## 这 90 天唯一应该追的 5 个数

只盯这 5 个：

1. `真实需求样本数`
2. `真实分发动作数`
3. `checklist 点击 / 领取`
4. `确认订阅与深度回复`
5. `更深版本 / SOP 的明确需求信号`

如果这 5 个数没有增长，就说明当前还不该扩产品线。

## 最后一句

未来 90 天，最重要的不是证明你“能做很多”，而是证明：

> 你已经找到一群在具体部署场景里真实会痛、真实会继续走、真实可能付钱的人。

只要这件事成立，后面的资产、SOP、支付、商城都会顺。
