# 市场发现冲刺计划

日期：`2026-05-24`

## 为什么要重置执行方式

当前项目不能继续默认从已有资产出发：

- 不能先假设 `开源 AI 项目部署前检查清单` 就是市场入口。
- 不能先假设 `开源 AI 项目上线与基础加固 SOP` 就是付费产品。
- 不能把 GitHub thread、文章点击、收藏、点赞自动解释成购买意图。

本轮冲刺的目标是把执行重心改成：

> 先找到真实市场里的重复痛点、替代方案缺口和购买动机，再决定写什么资产、卖什么形态、推进哪个 funnel。

| 角色 | 来源 | 在本项目里的用途 | 边界 |
| --- | --- | --- | --- |

| Trend Researcher | `product/product-trend-researcher.md` | 扫描候选市场、替代方案、社区问题、付费工具和趋势信号。 | 不用行业热度替代真实痛点证据。 |
| Feedback Synthesizer | `product/product-feedback-synthesizer.md` | 把 GitHub / Reddit / HN / X / 评论里的样本聚类，区分高频痛点与偶然问题。 | 不从单条样本直接推导产品方向。 |
| Reality Checker | `testing/testing-reality-checker.md` | 否决假阳性：曝光、点击、收藏、泛泛夸奖都不能算付费需求。 | 不能替代真实对话、报价、付款和交付证据。 |

暂不优先使用 `marketing/marketing-growth-hacker.md`。在还没确认稳定购买动机前，增长实验容易把注意力拉回“放大已有内容”，而不是“发现市场”。

## 本轮冲刺的判断问题

每个候选市场都必须回答：

1. 谁在什么具体时刻痛？
2. 他现在靠什么替代方案解决？
3. 替代方案为什么不够？
4. 这个问题造成了什么可感知代价？
5. 为什么现在要解决，而不是以后再说？
6. 他会为哪种形态继续一步：免费内容、模板、项目级判断、实施服务、长期托管？
7. 这个机会更像独立数字资产，还是更完整服务中的一个模块？
8. 用户当前已经为这个问题付出了什么成本：金钱、时间、人力、外包、客户信任、业务中断或风险敞口？
9. 这笔钱或时间原本从哪个预算 / 资源池里出？
10. 这个需求更像好奇、新效率、赚钱、避险，还是生存刚需？

## 候选市场池

本轮不只验证 deployment checklist 这一个方向。先并行观察 5 个候选市场：

| 候选市场 | 典型场景 | 可能付费触发 | 主要替代方案 |
| --- | --- | --- | --- |
| Self-hosted AI 部署可靠性 | Open WebUI、Ollama、Dify、Flowise、Langfuse、AnythingLLM 等从本地或 Docker 推向服务器。 | 本地能跑但服务器、网络、反代、权限、数据持久化、GPU、升级维护出问题。 | README、issue、Reddit、YouTube、ChatGPT、托管平台、DevOps 顾问。 |
| 自动化工作流上线与维护 | n8n、Make/Zapier 替代、自托管 webhook、凭据、队列、备份。 | 自动化开始影响真实业务，webhook / auth / 数据丢失会造成损失。 | 官方 docs、社区模板、云版 n8n、外包自动化顾问。 |
| AI Agent / 长期工作流落地 | 长任务、人工审批、Webhook 唤醒、持久状态、多 Agent 协作。 | demo 能跑，但不能稳定变成业务流程。 | ADK 文档、LangGraph、CrewAI、AutoGen、顾问、内部工程师。 |
| 小团队上线前风险判断 | 非专职运维把 GitHub 项目上线到 VPS / Docker / Caddy / Nginx / Supabase / Cloudflare。 | 需要对老板、客户或团队结果负责，害怕低级配置事故。 | 通用 AI、教程、朋友、issue、便宜外包。 |
| 本地 AI / 私有化工具给客户交付 | 帮客户部署本地知识库、Open WebUI、RAG、语音/视频处理、自动化后台。 | 交付失败会影响客户验收、尾款或后续维护。 | 自己摸索、临时脚本、ChatGPT、云服务、托管服务。 |

## 样本收集规则

每个样本必须写进：

- `docs/real-demand/demand-evidence-samples.csv`

最低字段必须覆盖：

- `source_url_or_person`
- `persona_guess`
- `user_quote`
- `scenario`
- `current_blocker`
- `feared_outcome`
- `current_workaround`
- `current_ai_tools_used`
- `current_ai_limitations`
- `requested_next_step`
- `willing_to_pay_or_book`
- `matched_offer_type`
- `notes`

如果缺少购买信号，`willing_to_pay_or_book` 必须写 `unknown` 或 `no_signal`，不能写 `possible`。

## 30 条样本门槛

在推进任何新付费资产前，先收集至少 30 条外部样本：

- Self-hosted AI：至少 8 条。
- 自动化工作流：至少 6 条。
- AI Agent / 长期工作流：至少 6 条。
- 小团队上线风险判断：至少 5 条。
- 客户交付型本地 AI / 私有化工具：至少 5 条。

每 10 条样本做一次聚类复盘：

- 高频问题是什么？
- 替代方案缺口是否重复？
- 是否出现明确代价？
- 是否出现“发项目给你看看”“能不能帮我判断”“多少钱”这类继续信号？
- 当前更像 `digital_asset`、`project_judgment`、`broader_service_module` 还是 `unclear`？
- 样本的 `economic_force` 分布是什么？候选市场排序是否应该按经济强制力调整，而不是只按样本数量、话题热度或内容可写性调整？

## 不能提前升级的事项

在达到下面证据前，不推进：

- 不新建完整商城或 checkout。
- 不把 SOP 当成已验证产品。
- 不把文章分发当成主要增长动作。
- 不把单条 GitHub / Reddit thread 当成 ICP。
- 不把通用 AI 能生成的内容包装成差异化资产。

## 可以马上做的事项

1. 扩充外部样本库。
2. 对每个候选市场补替代方案地图。
3. 用 Discovery Coach 视角重写访谈问题。
4. 做 3 个 `AI vs Frank 判断层` 对照实验。
5. 用 Reality Checker 审查每条样本是否真有购买动机。
6. 只在出现重复信号后，再写对应资产或报价草稿。

## 本轮输出标准

第一轮冲刺完成时，必须产出：

1. 至少 30 条真实样本。
2. 至少 5 类替代方案记录。
3. 至少 3 个明确的“现有方案不够”模式。
4. 1 个优先市场，最多 1 个备选市场。
5. 对当前 checklist / SOP 路线的判定：
   - 继续；
   - 改成项目级判断服务；
   - 并入更完整服务模块；
   - 暂停。
