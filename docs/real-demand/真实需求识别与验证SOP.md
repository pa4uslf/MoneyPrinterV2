# 真实需求识别与验证 SOP

日期：`2026-04-15`

这份 SOP 用来约束 `MoneyPrinterV2` 后续所有：

- 选题
- 资产设计
- landing page 文案
- 邮件跟进
- 付费资产判断

核心目标只有一个：

> 保证我们解决的是别人真实在疼、真实愿意继续走下去的问题，而不是自己脑中的“应该有需求”。

## 一、用《真需求》重写当前项目的判断框架

《真需求》里最重要的不是一句“用户需要什么”，而是这三件事：

1. `价值`
2. `共识`
3. `模式`

把它压到当前项目上，应该这样理解。

### 1. 价值：你到底在帮谁减少什么代价

当前阶段，真正有机会成立的价值不是：

- 教大家玩 AI
- 给大家更多工具资讯
- 给大家泛泛的部署教程

而是：

- 帮已经准备部署开源 AI / Web 项目的人，减少高代价试错
- 帮已经能折腾的人，更快看清顺序、暴露面和风险优先级
- 帮“能自己做，但不想被基础设施拖死”的人，收敛上线前的不确定性

### 2. 共识：为什么对方明明有痛，还没有行动

如果用户没有继续走 checklist、邮件、SOP，不一定是因为他没有需求。

更常见的原因是：

- 他没觉得你说的是他的具体场景
- 他没觉得这份清单会真的比公开文章更有执行价值
- 他没感觉到你理解他的“真实代价”
- 他还没相信你能带他从“会折腾”走到“可上线”

所以：

- 文章不是为了展示你知道很多
- checklist 不是为了多一个 lead magnet
- 邮件不是为了做增长动作

它们都是为了帮助用户形成一个共识：

> 这个人真的理解我现在卡住的问题，而且他给的是可以直接用的判断和顺序。

### 3. 模式：你最终收谁的钱，为什么能持续

当前项目的模式不该是：

- 先吸流量，再看看能不能接点服务

更不该是：

- 先做复杂商城，再找东西卖

当前最合理的模式是：

- 先用高意图内容吸到对的人
- 用 checklist 建立第一次关系
- 用重复出现的问题整理 SOP
- 用低价、低摩擦数字资产测试付费
- 只有在真实问题反复出现时，才考虑服务或更重产品

## 二、当前应该服务的真实人群

不要再用很大的泛词。

当前阶段，建议只服务下面三类人。

### 1. 自己会折腾，但怕上线事故的独立开发者

特征：

- 会看 README，会改环境变量，会拉仓库
- 能在本地跑起来
- 但对公网暴露、鉴权、备份、回滚、运维边界不够稳

他的真需求：

- 不是“学会部署”
- 而是“别让我在第一版上线时踩大坑”

他的付费触发器：

- 明确感受到时间成本和出错成本
- 发现自己不是不会，而是不知道先后顺序

### 2. 小团队里的兼职技术负责人

特征：

- 不是专职运维
- 但得负责把一个开源项目先挂起来
- 要对团队、老板或项目结果负责

他的真需求：

- 降低不确定性
- 提前发现明显风险
- 少一点“上线后再补”的侥幸

他的付费触发器：

- 害怕因为低级遗漏造成团队损失
- 愿意花小钱换确定性和结构

### 3. 正在筛 GitHub 项目的创业者 / builder

特征：

- 经常看到“这个项目很强”
- 但不确定它值不值得真的投入时间部署
- 容易被炫技功能吸引，忽视维护性和适配度

他的真需求：

- 不是“再看一个项目推荐清单”
- 而是“我到底该不该把这个项目推进到第一版可用”

他的付费触发器：

- 想节省选错方向的代价
- 想尽快从判断走到执行

## 三、当前不该优先服务的人

为了避免“印废纸机”，下面这些人群当前不要作为主目标：

- 纯 AI 热点围观者
- 只想看资讯不想动手的人
- 企业级复杂安全合规采购方
- 需要深度定制交付的大客户
- 完全不懂技术、也不准备自己动手的人

不是说他们永远不重要。

而是当前这条资产路径，不该为他们优化。

## 四、每个选题都必须过的 8 个问题

以后无论写文章、做清单、做 SOP，先回答下面 8 个问题。

### 1. 谁会在什么时刻需要它

不要写：

- “适合独立开发者”

要写成：

- “已经选了一个 GitHub 项目，准备部署到 VPS 或服务器，但还没想清楚公网暴露和配置风险的人”

### 2. 他现在最怕什么

例如：

- 上线后挂掉
- 密钥暴露
- 端口乱开
- 选错项目浪费一周
- 本地能跑但服务器一直起不来

### 3. 他现在怎么解决

例如：

- 反复试 README
- 到处翻 issue
- 问朋友
- 把一堆博客收藏起来
- 硬上服务器再出错

### 4. 现有解决方式为什么不够

例如：

- 太碎
- 太泛
- 只讲“怎么做”，不讲“先做什么”
- 只讲功能，不讲暴露面和回滚
- 看起来多，实际没有顺序感

### 5. 这份内容到底给他带来什么确定性

例如：

- 帮他先判断值不值得部署
- 帮他把上线前检查项按结构过一遍
- 帮他知道哪些风险必须先收掉

### 6. 它更偏功能价值，还是更偏保障感

如果两者都没有，这个题就弱。

当前优先做：

- 高功能价值
- 高保障感
- 高场景强度

### 7. 它自然承接到哪里

允许的自然承接只有：

- 文章 -> checklist
- checklist -> welcome / follow-up
- checklist / follow-up -> SOP

如果一个题很难自然承接，就先别做。

### 8. 用户会用什么语言复述这个问题

以后标题和首屏尽量贴近用户脑中的原话，例如：

- “本地能跑，为什么服务器就是不行”
- “这个 GitHub 项目到底值不值得部署”
- “上线前哪些必须先做，哪些可以后补”

## 五、当前内容和资产的表达规则

### 1. 不要把自己写成“老师”，要写成“判断层”

用户需要的不是再来一个泛教程作者。

他们需要的是：

- 帮他们减少误判
- 帮他们少走大代价弯路
- 帮他们看顺序

### 2. 不要只卖功能，也要卖保障感

目前最有价值的表达，不是：

- “我会教你怎么部署”

而是：

- “我帮你在真正投入时间和成本前，先把最容易导致返工和暴露的问题看清楚”

### 3. 不要把用户写成小白

当前主用户很多不是零基础。

他们往往已经：

- 会看文档
- 会起服务
- 会查 issue

但他们卡在：

- 判断
- 顺序
- 暴露面
- 稳定性

### 4. 不要把内容写成百科，要写成动作前判断

优先写：

- 值不值得做
- 先做什么
- 先查什么
- 哪些必须先处理
- 哪些可以后补

## 六、当前验证证据的最小门槛

在推进付费 SOP 前，至少要看到下面这些证据中的大部分：

### 1. 流量层证据

- 至少 3 到 5 个真实外部分发动作已执行
- 至少 2 篇文章被真实投放到问题现场
- 有明确的文章页访问和 checklist 页访问

### 2. 转化层证据

- checklist 有真实点击
- checklist 有真实领取 / unlock
- 有确认订阅，而不是只有表单提交

### 3. 共识层证据

- 有真实回复、邮件、DM、评论
- 有人追问更完整版本
- 有人问“先处理什么”“有没有更完整 SOP”

### 4. 付费前证据

- 同类问题反复出现 3 次以上
- 免费清单无法完整承接这些问题
- 用户愿意继续往更深执行层走

如果这些证据不出现，不要急着推 SOP。

## 七、当前最应该怎么收集真需求

不要再只靠内部推演。

当前最该做的是两条线并行。

### 线 A：公开问题现场

从下面这些地方收集真实表达：

- GitHub issues / discussions
- Reddit
- V2EX
- 知乎
- 自己收到的邮件、私信、评论

重点不是收集“主题”，而是收集：

- 原话
- 场景
- 出错代价
- 当前 workaround
- 是否愿意继续行动

### 线 B：已有关系链

去找已经在折腾部署的人，直接问：

1. 你最近在部署什么
2. 最卡的三个点是什么
3. 你最怕出什么事故
4. 你现在怎么判断一个项目值不值得部署
5. 如果有一份结构化 checklist / SOP，哪部分最值得你花钱

## 八、当前最重要的执行纪律

- 每新增一个选题，先写清楚对应场景，不准只写“大方向”
- 每新增一个资产，先写清楚它替代了用户哪种低效 workaround
- 每次想扩主题前，先看 deployment 主题簇是否已经出现重复证据
- 每次想推进付费前，先看 checklist 是否已经跑出真实深度需求

## 九、最后一句

如果以后你对某个题目拿不准，先问一句：

> 这是别人已经在付出时间、焦虑、返工成本的问题，还是只是我觉得“这个内容应该有用”？

只有前一种，才值得进当前主线。
